# JSON Sync Implementation Summary

## What is JSON Sync?

A simplified architecture for the agenda system where:
- **Mobile app** writes state changes as JSON files to a `sync/` directory
- **Emacs** processes these JSON files and updates the actual source org files
- **Syncthing** handles bidirectional file synchronization

## The Problem We're Solving

### Current Architecture (v0.8.0)
Mobile app directly edits org files when you change a TODO state:
1. User taps TODO item
2. App reads org file
3. App parses file structure
4. App updates headline in memory
5. App writes entire file back
6. App re-syncs database

**Issues:**
- 🔴 Race conditions when mobile and Emacs both edit
- 🔴 Complex conflict resolution code (~800 lines)
- 🔴 Risk of corrupting org files if bugs in parser/writer
- 🔴 No audit trail of mobile changes
- 🔴 Hard to debug when things go wrong

### New Architecture (v0.9.0)
Mobile app writes simple JSON files, Emacs does all org file editing:
1. User taps TODO item
2. App updates local database (optimistic UI)
3. App writes JSON: `sync/ABC_DONE_123456.json`
4. Syncthing syncs JSON to desktop
5. User runs `M-m s s` in Emacs
6. Emacs updates source files and deletes JSONs
7. Emacs exports updated `agenda.org`
8. Mobile detects change and refreshes display

**Benefits:**
- ✅ No race conditions (Emacs is single writer)
- ✅ Simpler mobile code (~800 lines removed)
- ✅ Safer (JSON can't corrupt org files)
- ✅ Clear audit trail (JSON files with timestamps)
- ✅ Easier to debug (inspect JSON files)
- ✅ Survives mobile app crashes

## Key Concepts

### ORIGIN_ID Property
Each agenda item has a unique identifier in its properties:
```org
* TODO Morning Meditation
:PROPERTIES:
:ORIGIN_ID: F85F4DCD-9FA2-4022-B624-C390D7F4822C
:END:
```

This ID is used to:
- Match mobile database records to org headlines
- Track items across mobile/desktop edits
- Identify which headline to update from JSON

**Migration:** App will support `ORIGIN_ID` (new), `SOURCE_ID` (old), and `ID` (legacy) for backwards compatibility.

### JSON Sync Files
When you change a TODO state on mobile, it writes:

**Filename:** `sync/F85F4DCD_DONE_1709467200000.json`

**Content:**
```json
{
  "id": "F85F4DCD-9FA2-4022-B624-C390D7F4822C",
  "status": "DONE",
  "changed_at": "2026-03-02T12:35:00-05:00"
}
```

**Filename format:** `<origin-id>_<new-state>_<timestamp>.json`

### Sync Directory Structure
```
<org-root>/
├── agenda.org                    # Generated by Emacs (M-m s e)
├── inbox.org                     # Your source files
├── Brain/
│   └── tasks.org
├── phone_inbox/                  # Mobile quick capture
│   └── *.org
└── sync/                         # NEW - JSON state changes
    ├── ABC_TODO_123.json         # Pending changes
    ├── DEF_DONE_456.json
    └── failed/                   # Failed syncs
        ├── XYZ_TODO_789.json
        └── XYZ_TODO_789.error    # Error details
```

## User Workflow

### Before (Direct Write)
```
Mobile: Change TODO → Update org file → Hope Emacs didn't also edit
```

### After (JSON Sync)
```
Mobile: Change TODO → Write JSON → Done!
Desktop: M-m s s (periodically) → Process all pending changes
```

## Emacs Commands

Two new commands for Emacs:

1. **`M-m s s`** (Sync)
   - Reads all JSON files in `sync/`
   - Updates source org files
   - Deletes processed JSONs
   - Moves failed syncs to `sync/failed/`

2. **`M-m s e`** (Export)
   - Generates `agenda.org` with ORIGIN_ID properties
   - Mobile reads this file to display agenda

## Migration for Users

### Prerequisites
1. Emacs configuration updated with JSON sync functions
2. Syncthing running and syncing org files
3. Backup of existing org files

### Steps
1. **Update mobile app** to v0.9.0
2. **Update Emacs config** with M-m s s and M-m s e functions
3. **Run M-m s e** to generate agenda.org with ORIGIN_ID
4. **Test state change** on mobile
5. **Run M-m s s** in Emacs to process
6. **Verify** change reflected in source file and mobile display

### What Can Go Wrong?

| Issue | Fix |
|-------|-----|
| "Pending syncs" badge won't clear | Run `M-m s s` in Emacs |
| State changes don't appear after refresh | Check if JSON was processed (deleted from sync/) |
| JSON file in sync/failed/ | Read .error file, manually fix in Emacs |
| "Cannot write to sync directory" | Recreate storage access in mobile Settings |

## Key Decisions Made

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Property name | ORIGIN_ID (not SOURCE_ID) | Clear distinction for new format |
| Filename | `<id>_<state>_<timestamp>.json` | Sortable, human-readable, unique |
| Directory | `<org-root>/sync/` | Simple, alongside other org files |
| Update strategy | Optimistic (immediate UI) | Better UX, rollback on failure |
| Error handling | Move to sync/failed/ | Audit trail, user can debug |
| Toggl timing | Immediate on mobile | Responsive, doesn't wait for Emacs |

## Risk Mitigation

| Risk | Mitigation |
|------|------------|
| JSON files accumulate | Show badge, add debug screen, clear instructions |
| Optimistic updates confuse users | Clear error messages, rollback on failure |
| Sync directory permission issues | Test on startup, helpful error messages |
| Breaking change for users | Backwards compatible, phased rollout |
| Performance with many syncs | Tested up to 1000 JSONs |
