## Emacs Processing Flow

```
┌────────────────────────────────────────────────────────────────────┐
│                    Emacs Sync Processing (M-m s s)                  │
└────────────────────────────────────────────────────────────────────┘

USER RUNS: M-m s s
     │
     ▼
╔═══════════════════════════════════════════════════════════════════╗
║ 1. SCAN sync/ DIRECTORY                                            ║
╚═══════════════════════════════════════════════════════════════════╝
     │
     ├─ Find: ABC_DONE_1709467200000.json
     ├─ Find: DEF_TODO_1709467201234.json
     └─ Find: XYZ_IN-PROGRESS_1709467205678.json
     │
     ▼
╔═══════════════════════════════════════════════════════════════════╗
║ 2. PROCESS EACH JSON FILE (in timestamp order)                    ║
╚═══════════════════════════════════════════════════════════════════╝
     │
     │ For ABC_DONE_1709467200000.json:
     │
     ├─ Parse JSON → { "id": "ABC", "status": "DONE", ... }
     │
     ├─ Find headline with ORIGIN_ID = "ABC" in source files
     │
     ├─ Update headline TODO state → DONE
     │
     ├─ Add CLOSED timestamp (if DONE state)
     │
     ├─ Save source file
     │
     ├─ Delete ABC_DONE_1709467200000.json
     │
     └─ Log: "Processed ABC → DONE"
     │
     │ (Repeat for other JSON files)
     │
     ▼
╔═══════════════════════════════════════════════════════════════════╗
║ 3. ERROR HANDLING                                                  ║
╚═══════════════════════════════════════════════════════════════════╝
     │
     │ If ORIGIN_ID not found:
     │
     ├─ Move JSON to sync/failed/
     │
     ├─ Create .error file with details:
     │    "Origin ID ABC not found in any org file"
     │
     └─ Log error
     │
     ▼
╔═══════════════════════════════════════════════════════════════════╗
║ 4. DONE                                                            ║
║ Message: "Processed 2 state changes, 1 failed"                    ║
╚═══════════════════════════════════════════════════════════════════╝

USER RUNS: M-m s e (Export agenda)
     │
     ▼
╔═══════════════════════════════════════════════════════════════════╗
║ GENERATE agenda.org with updated states                           ║
║                                                                    ║
║ * TODO Morning Meditation                                         ║
║ SCHEDULED: <2026-03-02 Mon 05:00>                                ║
║ :PROPERTIES:                                                      ║
║ :ORIGIN_ID: ABC                                                   ║
║ :END:                                                             ║
║                                                                    ║
║ * DONE Lunch                     ← Updated from JSON!            ║
║ SCHEDULED: <2026-03-02 Mon 12:00>                                ║
║ CLOSED: [2026-03-02 Mon 12:35]   ← Added by Emacs               ║
║ :PROPERTIES:                                                      ║
║ :ORIGIN_ID: DEF                                                   ║
║ :END:                                                             ║
╚═══════════════════════════════════════════════════════════════════╝
     │
     │ Syncthing syncs agenda.org to mobile
     ▼
```

## Error Handling Flow

```
┌─────────────────────────────────────────────────────────────────┐
│                     Error Scenarios                              │
└─────────────────────────────────────────────────────────────────┘

SCENARIO 1: Write Failure
━━━━━━━━━━━━━━━━━━━━━━━━
User taps TODO
  ↓
updateTodoState()
  ↓
Update DB ✓ (optimistic)
  ↓
Write JSON ✗ (permission denied)
  ↓
ROLLBACK: noteDao.updateState(noteId, oldState)
  ↓
Show error: "Cannot write to sync directory"
  ↓
UI reverts to old state


SCENARIO 2: Emacs Can't Find ORIGIN_ID
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Mobile writes: ABC_DONE_123.json
  ↓
Syncthing syncs
  ↓
Emacs M-m s s
  ↓
Search for ORIGIN_ID="ABC" → NOT FOUND
  ↓
Move to: sync/failed/ABC_DONE_123.json
Create:  sync/failed/ABC_DONE_123.error
         "Origin ID ABC not found in any org file"
  ↓
Mobile refresh
  ↓
Detect .error file (optional future feature)
  ↓
Show warning: "Some syncs failed, check Emacs"


SCENARIO 3: Offline Mode
━━━━━━━━━━━━━━━━━━━━━━━━
Mobile: Change 5 items
  ↓
Write 5 JSON files ✓
  ↓
Syncthing offline ✗
  ↓
Mobile shows: "Pending syncs: 5"
  ↓
User brings laptop online
  ↓
Syncthing syncs all 5 JSONs
  ↓
User runs M-m s s
  ↓
All 5 processed ✓
  ↓
Badge clears automatically
```

## Benefits Visualization

```
┌─────────────────────────────────────────────────────────────────┐
│                Before (Direct File Write)                        │
└─────────────────────────────────────────────────────────────────┘

Mobile app writes directly to source org files
  ↓
Problems:
  • Race conditions (mobile vs Emacs edits)
  • Complex conflict resolution
  • Corrupted org files if parser/writer bugs
  • ~800 lines of file manipulation code
  • No audit trail
  • Hard to debug

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

┌─────────────────────────────────────────────────────────────────┐
│                 After (JSON Sync)                                │
└─────────────────────────────────────────────────────────────────┘

Mobile writes JSON → Emacs updates source files
  ↓
Benefits:
  ✓ No race conditions (Emacs is single writer)
  ✓ Simple conflict resolution (Emacs decides)
  ✓ Safe (JSON can't corrupt org files)
  ✓ ~800 lines removed (simpler code)
  ✓ Clear audit trail (JSON files with timestamps)
  ✓ Easy to debug (inspect JSON files)
  ✓ Survives mobile app crashes
  ✓ Can batch process multiple changes
```
