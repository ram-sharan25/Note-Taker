# System Quirks & Non-Obvious Behaviors

This document tracks technical "quirks," edge cases, and non-obvious design choices that are essential for developers and users to understand.

---

## QRK-01: `quick.org` Direct Editing

**Context:** The app normally uses the [JSON Sync Workflow](FLOWS.md#3-todo-state-update-flow-json-sync) for status changes to avoid race conditions with Emacs.

**The Quirk:** Status changes in `phone_inbox/quick.org` use **Direct File Editing** instead of JSON Sync.

**Why?**
1. **Ownership:** The mobile app is the primary "owner" of `quick.org`. Emacs only processes this file occasionally during refiling.
2. **Speed:** Direct editing allows for instantaneous updates on the phone without waiting for a sync-loop back from Emacs.
3. **Complexity:** Since Emacs doesn't "watch" `quick.org` for status changes, writing a JSON for it would lead to orphaned sync files.

## QRK-02: Pending Sync Layering (intent-over-file)

**Context:** `agenda.org` is a view generated by Emacs. When Syncthing pulls a new version of `agenda.org` to the phone, it might be "stale" (it doesn't include the phone's most recent status changes yet).

**The Quirk:** During the file re-sync process, the app **lays its own pending JSON changes on top** of the data imported from the file.

**Why?**
- To prevent the "Stale File Reversion" flaw where a status you just marked "Done" on your phone suddenly flips back to "TODO" after a sync.
- The app trusts its own `sync/*.json` files more than the incoming `agenda.org` file.
- *See [Research: Pending Sync Layering](research/PENDING_SYNC_LAYERING.md) for details.*

## QRK-03: Lowercase UUID Handling in JSON Sync

**Context:** The mobile app uses UUIDs to identify tasks. Emacs' `org-id` system also uses UUIDs.

**The Quirk:** Android generates **lowercase hex UUIDs** (e.g., `a1b2...`), while some Emacs configurations might use uppercase.

**Why it matters:** 
- The initial JSON sync regex only accepted uppercase hex. This caused the app to silently fail all status updates because it would write a lowercase filename that failed validation.
- **Fix (v0.9.0):** The `isValidSyncFilename` regex now accepts `[A-Fa-f0-9]` to ensure robustness across platforms.

## QRK-04: Audio Focus Transient Gain

**Context:** Voice recognition requires a "hot" microphone. When the recognizer restarts (every 150ms during continuous listening), there is a gap.

**The Quirk:** We request `AUDIOFOCUS_GAIN_TRANSIENT` when the mic starts and **do not release it** until the entire session is over (even during the restart gaps).

**Why?**
- To prevent audiobook or music apps from thinking the app is "done" during the 150ms gap. If focus were released and re-acquired, other apps would "blip" or start playing again during the silence.
- This ensures a seamless, silent listening experience even as the underlying Android recognizer resets.

## QRK-05: SHA-256 Hash-Based File Sync Optimization

**Context:** Large org files (1000+ headlines) are expensive to parse (200-500ms).

**The Quirk:** The app stores a SHA-256 hash of every agenda file. On every sync attempt, it checks the file's current hash against the stored hash.

**Why?**
- To skip parsing unchanged files. If the hash matches, the app does nothing.
- This drastically reduces CPU usage and battery consumption during background syncs via Syncthing.

---

## QRK-06: Case-Insensitive Extension Matching

**Context:** Users might name their files `.org`, `.ORG`, or `.Org`.

**The Quirk:** The app uses case-insensitive matching for file extensions and TODO keywords.

**Why?**
- To avoid "Missing File" bugs on platforms where file extensions might be inconsistently cased (especially when syncing via Syncthing from multiple different OSes).
