# Research: Agenda Status Reversion Analysis

## The "Generated View" Flaw

The current agenda architecture (based on ADR 005) relies on a single `agenda.org` file as the source of truth for the mobile UI. However, this creates a critical flaw in bidirectional synchronization.

### 1. The Conflict
- **Source of Truth:** The actual tasks live in various source files on the desktop (e.g., `tasks.org`, `projects.org`).
- **The View:** `agenda.org` is a **disposable view** generated by Emacs by aggregating these source files.
- **The Loop:**
    1. Mobile app changes a status and writes a JSON "Change Request" to `sync/`.
    2. Syncthing syncs a stale `agenda.org` from the desktop to the phone (before Emacs has processed the JSON).
    3. The mobile app detects the file change and re-parses `agenda.org`.
    4. **The Reversion:** The app's `syncFileToDatabase` wipes the local database and imports the stale "TODO" state from the file, overwriting the user's "Done" intent.

### 2. The Root Cause
The mobile app treats `agenda.org` as the **Absolute Truth**, whereas it is actually just an **Update Stream**. The app currently has no "memory" of its own pending intents once a new file arrives.

---

## The Solution: Pending Sync Layering

To fix this, the mobile app must transition from "Trusting the File" to **"Trusting the Intent."**

### Layered Reality Model
We must implement a layering logic during the file sync process:

1.  **Layer 1 (The File):** Import the data from `agenda.org` into the database as the "Base State."
2.  **Layer 2 (The Intent):** Immediately scan the `sync/` directory for any JSON files written by the phone that haven't been deleted yet.
3.  **The Result:** "Re-apply" those pending statuses over the database records. 

**Logic:**
> If (Task ID exists in `sync/*.json`) 
>    THEN (Status in Database = Status in JSON)
>    ELSE (Status in Database = Status in `agenda.org`)

### Benefits
- **Perceptual Consistency:** The user sees their change "stick" immediately and stay there, even if Syncthing brings in stale files.
- **Robustness:** It handles the lag between "Mobile Tap" and "Emacs Processing" without confusing the user.
- **Simplicity:** It reuses the existing `sync/` folder as a local cache of "Truth."

---

## Workflow Implications

### Mobile Side
- **Non-Destructive Sync:** `syncFileToDatabase` must become a two-stage process: *Import* followed by *Layer*.
- **ID Stability:** Relies entirely on `ORIGIN_ID`. Since every task from Emacs is from the pipeline and contains an ID, this is handled.

### Desktop Side
- Emacs remains the single writer for source files.
- The `agenda.org` export remains a "Mirror," but the mobile app is now smart enough to ignore the mirror when it knows it has a more recent change pending.

### The `quick.org` Exception
- `phone_inbox/quick.org` continues to use **Direct Editing** because the mobile app is the primary owner of that file until it is refiled on the desktop.
